# 💻 [2026-01] 나만의 C언어 소프트웨어 개발 프로젝트

## 1. 시나리오 제목
* 초보 편의점 알바생을 위한 POS(판매 시점 관리) 시뮬레이터

<br>


## 2. 시나리오 (5~10줄)
* 이 프로그램은 편의점 아르바이트를 처음 시작한 초보 알바생이 계산 실수를 줄이고 매장을 효율적으로 관리할 수 있도록 돕는 **'통합 POS 관리 시스템'** 입니다.
* 사용자는 판매할 상품명, 가격, 수량을 입력하여 총 결제 금액을 산출하고, 손님에게 바든 금액에 따른 거스름돈을 정확히 계산할 수 있습니다. 시스템은 향후 업데이트를 통해 1+1 행사 상품 여부를 판별하거나 폐기 상품 발생 시 알람을 주는 기능을 추가할 예정입니다. 또한, '영업 종료' 전까지 반복해서 손님을 응대하고, 하루 동안 판매된 물건 리스트를 배열에 저장하여 총 매출을 자동으로 정산하는 기능까지 구현하고자 합니다. 최종적으로는 실제 근무 현장에서 겪을 수 있는 상황들을 하나씩 기능으로 구현해 보며, 실수 없이 업무를 처리할 수 있는 나만의 스마트한 업무 보조 도구를 완성해 나갈 계획입니다.

<br>


## 3. 예상 기능 및 메뉴 (최소 5개)
1. 상품 정보 입력 및 결제: 판매 상품의 가격과 수량 입렵
2. 거스름돈 자동 계산: 받은 돈 대비 잔돈 산출
3. 행사 및 할인 적용: 1+1, 2+1 상품 판별 및 할인 기능
4. 일일 매출 정산: 판매 데이터 누적 및 총합 계산
5. 재고 현황 조회: 현재 남은 상품 수량 확인
6. 시스템 종료: POS 프로그램 종료 및 정산 결과 출력

## 🟦 [1차 과제: V1.0] 기초 뼈대 구축 (입출력과 변수)
### **✨V1.0 개발 내용:**
  * 내용: 사용자로부터 판매할 상품명(문자), 단가(정수), 판매 수량(정수), 받은 금액(정수)을 입력받아 총 결제 금액과 거스름돈을 출력하여 계산합니다.
  * 구현사항: 상품명(`char`), 단가/수량/받은 돈(`int`), 부가세율(`dluble`)등 3개 이상의 자료형을 사용하여 기초 POS화면을 구현했습니다.
  * 연산: `단가 * 수량` 을 통한 총액 계산과 `받은 돈 - 총액`을 통한 거스름돈 산출로직을 적용했습니다.

    * **적용 내용:** 단순한 계산기를 넘어 '기초 결제(1차) ➔ 1+1 행사 및 할인 판별(2차) ➔ 반복적인 손님 응대 메뉴(3차) ➔ 일일 총 매출 정산 및 재고 관리(4차)'로 이어지는 실무 중심의 로드맵을 확립함.
      
 2. **내용 2**
    * **프롬프트 요약:** "1차 과제 필수 요건(변수 5개, 자료형 3개 이상)을 충족하면서 실제 편의점 결제 로직에 필요한 데이터 구조 논의"
    * **적용 내용:** 상품명(`char`), 가격/수량(`int`) 외에 부가세(10%) 계산을 위한 실수형(`double`) 자료형을 추가하여 '자료형 3개 이상' 조건을 충족하고, 실제 영수증과 유사한 데이터 구조를 설계함. 
    
### **🛠️ Troubleshooting & 기술 회고:**
  1. **문제 1:** 상품 이름(문자열)을 입력받을 때 `scanf("%s, item);`에서 띄어쓰기가 포함된 상품명(예: 삼각 김밥)입력 시 오류가 발생함.
    * **해결:** AI에게 질문하여 `%s`는 공백 전까지만 읽다는다는 것을 배우고, 이번 단게에서는 공백 없는 상품명을 사용하거나 향후 `fgest`함수 등을 공부하여 보완하기로 함.
  2. **문제 2:** 결제 금액을 출력할 때 숫자가 너무 붙어있어서 영수증 같은 느낌이 나지 않음.
    * **해결:** `printf` 서식 지정자 사이의 간격을 조정하고 `--- 영수증 ---`과 같은 구분선을 추가하여 가독성을 높이고 UI 디자인을 적용함.
  3. **문제 3:** 거스름돈 계산 시 받은 돈이 총액보다 적을 경우 마이너스(-) 값이 출력되는 현상 발생.
    * **해결:** 현재 단계에서는 조건문(`if`)을 사용할 수 없으므로, 사용자(알바생)가 반드시 총액보다 큰 금액을 입력한다는 가정하에 진행함. 2차 과제에서 조건문을 배워 이 문제를 해결할 계획임.

## 🟩 [2차 과제: V2.0] 조건 분기 적용 (조건문)
### **✨V2.0 개발 내용:**
  * 내용: 1차 과제의 기본 틀에 `if else` 조건문을 도입하여, 실제 편의점에서 알바생이 판단해야 하는 4가지 핵심 상황을 자동으로 처리하는 지능형 POS 시스템을 구축했습니다.
  * 구현사항:
    1. 성인 인증 시스템: 주류/담배 상품 번호 선택 시 구매자 나이를 입력받아 20세 미만은 판매를 자동 차단함(`if else` 중첩 활용).
    2. 폐기 상품 판별: '도시락' 상품 번호와 '현재 시간'을 `&&` 연산자로 동시에 체크하여 22시 이후일 경우 판매 불가 메시지 출력 및 프로그램 종료.
    3. 봉투 유상 제공 로직: 결제 전 봉투 필요 여부(1/0)를 입력받아 선택 시 총 결제 금액에 100원을 자동으로 가산함 (`+=` 연산 활용).
    4. 실시간 잔액 확인: 손님이 낸 돈과 최종 금액을 비교하여, 잔액이 부족하면 부족한 금액을 알려주고 결제를 승인하지 않음 (`if else` 활용).
  * 연산
    * 논리 연산(`&&`): 상품 번호와 시간이라는 두 가지 조건을 동시에 체크하여 폐기 여부를 판별하는 로직을 적용했습니다.
    * 복합 대입 연산(`+=`): 선택 사항인 봉투값(100원)을 기존 총액에 간결하에 합산하여 계산했습니다.
    * 비교 연산(`>=`, `<`, `==`): 성인 여부 판별, 결제 가능 잔액 확인, 상품 종류 식별을 위한 크기 및 일치 비교를 수행했습니다.
    * 차액 산출: `받은 돈 - 총 결제 금액`을 통해 거스름돈 또는 부족 금액을 정확히 계산했습니다.
    
### **🤖 AI 파트너십 과정**
 1. **내용 1**
    * **프롬프트 요약:**  "편의점 알바 경험을 코드로 옮기기 위해 성인인증, 폐기 확인 등 조건문이 필요한 실무 시나리오를 구성했는데, C언어 기초 단계에서 구현 가능한지 논리적 타당성 확인 요청"
    * **적용 내용:** 직접 구상한 4가지 상황(나이, 시간, 봉투, 잔액)을 `if-else` 구조로 배치하는 과정에서, 판매 불가 상황 시 `return 0;`을 사용하여 결제 단게 진입을 차단하는 방어적 설계 아이디어를 AI와 협력하여 구체화함.
 2. **내용 2**
    * **프롬프트 요약:** "아직 배우지 않은 복잡한 기능을 제외하고, 현제까지 배운 '정수 비교'와 '조건문'만으로 특정 상품(도시락, 주류 등)을 식별할 수 있는 가장 효율적인 방법 논의"
    *  **적용 내용:** 문자열 대신 상품 카테고리에 고유 번호를 부여하는 방식을 도출하여, 배운 범위 내에서 정직하고 논리적으로 문제를 해결함.
    
### **🛠️ Troubleshooting & 기술 회고:**
  1. **문제 1:** 복합 조건식(`&&`)의 논리 범위 설정 오류
     * **증상:** 폐기 로직 작성 시, `if (hour >= 22) 만 설정하여 도시락이 아닌 다른 상품을 결제할 때도 밤 10시가 넘으면 판매가 중단되는 논리적 오류 발생.
     * **해결:** 논리 연산자(`&&`)를 사용하여 `(itemCode == 1 && hour >= 22)`와 같이 '상품 번호'와 '시간'이 동시에 일치할 때만 작동하도록 조건을 정교화하여 해결함. 이를 통해 복합 조건 설정 시각 조건의 독립성과 결합성을 고려하는 법을 익힘.
  2. **문제 2:** 관계 연산자 표기 오류 및 구문 에러
     * **증상:** '같지 않다'를 표현할 때 `!>=`와 같이 존재하지 않는 연산자 조합을 사용하여 컴파일 에러 발생.
     * **해결:** C언어 표준 연산자인 `!=`와 `>=`의 정확한 표기법을 재확읺마. 논리 부정 연산자(`!`)는 단독으로 쓰이거나 `!=`형태로 쓰인다는 문법적 규칙을 다시 학습하여 수정함.
  3. **문제 3:** 개발 환경(IDE)의 입력 모드 간섭 해결
     * **증상:** 코드 수정 중 새로운 글자를 입력하면 뒷부분의 코드가 자꾸 지워지며 덮어씌워지는 현상으로 인해 작업 흐름이 끊김.
     * **해결:** 키보드의 `Insert` 키가 눌려 '덮어쓰기 모드'가 활성화 되었음을 AI 파트너를 통해 발견함. 다시 `Insert`를 눌러 '삽입 모드'로 전환하여 해결하였으며, 개발 환경 하단의 상태 표시줄을 확인해 모드를 체크하는 습관을 갖게 됨.
  4. **기술 회고:**
     * 1차 과제에서 해결하지 못했던 '잔액 부족 시 음수 출력' 문제를 조건문 분기 처리를 통해 완벽히 해결하며 프로그램의 안정성을 높였습니다.
     * 특히 '정수 비교' 방식을 응용해 로직을 구현하면서 조건문의 원리와 프로그램의 제어 흐름을 더 깊이 있게 이해할 수 있었습니다. 단순 구현보다 '배운 것을 어떻게 활용할 것인가'에 대한 고민이 중요함을 깨달은 단계였습니다.
